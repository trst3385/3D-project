몬스터 프리팹에 World Space. UI 체력바를 부착하고, 실제 체력 데이터와 연동

- 구조 설계: UI Canvas를 World Space 모드로 설정하여 몬스터 자식으로 배치.
- 자동 연결: transform.Find를 사용하여 인스펙터 의존성 없이 런타임에 체력바를 찾도록 구현.
- 데이터 연동: 몬스터 체력 스크립트의 TakeDamage() 함수 내부에서 Slider.value를 실시간 업데이트하여 체력 감소 시각화 성공.

- 발생했던 러블슈팅:
   1. 초기값 오류: Awake() 호출 순서와 Start의 초기화 시점이 맞지 않아 발생한 문제. → Awake()에서 체력을 초기화 하지 않고 Start()에서 체력을 초기화한 후 슬라이더 값을 갱신하는 방식으로 수정
   2. 비율 문제: 체력바 Slider의 Max Value가 기본값(1)으로 고정되어 있어 몬스터의 체력값 10과 매칭되지 않음. → 인스펙터와 Awake에서 MaxValue = maxHealth로 강제 지정하여 해결
   3. UI 표현 문제: Fill Area와 Fill 오브젝트의 Anchor가 Stretch로 되어있지 않아 체력바가 비정상적으로 깎임. → Anchor를 Stretch로, Offset을 모두 0으로 재설정하여 해결

- 다음 단계: 
   1. 직접 구현해 보았기에 나중에 에셋 스토어의 체력바 에셋을 쓰더라도, 어떤 컴포넌트를 조작해야 하는지 훨씬 빨리 이해할 수 있을거야
   2. 유니티 UI 시스템의 원리: 슬라이더의 Max Value와 Fill Area의 Anchor 설정은 UI 시스템의 기본 중 기본이야, 오늘 이를 통해 UI 렌더링 원리를 확실히 깨달았지
   3. 지금은 몬스터 오브젝트의 자식으로 체력바 UI가 들어가 있지만 나중에 개발 규모가 커질때 다른 방식 혹은 옵저버 패턴으로 고려해보자

기존까지 하드코딩된 모든 스크립트들을 데이터 중심 설계(Data-Driven Design)로 리팩토링, ScriptableObject(SO)를 활용해 데이터와 로직을 완전히 분리.
타워의 사거리 설정 및 관리 방식의 개선(SphereCollider의 Radius 방식에서 SO의 range 변수값으로 변경)

- 데이터 관리: EnemyData, TowerData, PlayerData를 각각 ScriptableObject로 생성.
- 특히 몬스터(빨간 큐브)는 EnemyMovement -> NormalEnemy으로 이름 변경. 이 스크립트 중심축으로 삼아 몬스터의 데이터를 관리.
   1. NormalEnemy 스크립트에 EnemyData So를 연결
   2. EnemyHealth 스크립트는 NormalEnemy에 연결된 So를 받아 체력값 설정
- 나무(AttackTower) 및 발사체(Bullet) 데이터 적용:
   1. TowerData SO를 생성하여 타워 자체 값 외에도 발사체의 속도, 사거리, 발사체 스탯 등을 통합 관리
   2. 다만, 발사체(Bullet)는 타워와 물리적으로 분리된 별개의 오브젝트라서 프로퍼티 참조 방식이 아닌 '생성 시점의 데이터 주입' 방식을 채택
   3. AttackTower가 Instantiate 직후 Bullet.Initialize() 함수를 통해 타워가 가진 SO 데이터를 직접 전달함으로써, 발사체가 타워의 생명주기에 의존하지 않고 독립적으로 동작하도록 설계
   4. 발사체의 결합도(Coupling)를 낮추고, 만약에 타워가 파괴 되어도 발사체가 안전하게 동작할 수 있는 구조를 확보
   5. 즉, 타워와 발사체는 별개의 오브젝트지만 몬스터가 사거리 안에 들어오면 발사체는 생성되자마자 독자적으로 값 받은 상태로 동작하니까 타워의 **생명주기**에 대한 의존도 없이 독자적으로 행동

**생명주기(Life Cycle)란?: 용어는 프로그래밍, 특히 유니티 같은 게임 엔진에서 아주 중요한 개념이야. 쉽게 말해 "오브젝트가 태어나서부터 죽을 때까지 겪는 일련의 과정"을 뜻해**
**즉, 생명주기가 서로 다른 오브젝트들이 서로의 존재에 의존하지 않게 만드는 것(독립성), 이게 바로 프로그래밍에서 말하는 '안전한 설계'의 핵심이야**


- 기존에는 인스펙터에서 SphereCollider의 Radius 값을 직접 수정해서 사거리를 조정했지만 SO를 도입했기에 TowerData SO에 range 변수를 별도로 선언.
- 우선순위 결정 및 자동화:
   1. AttackTower 스크립트의 Start() 함수에서 col.radius = towerData.range; 코드를 호출함으로써, 에디터 상의 Radius 값보다 SO의 range 변수가 항상 우선하도록 강제
   2. 이를 통해서 인스펙터에서의 Radius는 개발자가 타워의 사거리를 시각적으로 확인하기 위한 '가이드라인'으로 활용하고, 실제 게임 로직상의 사거리는 SO 파일 하나로 통합 관리하는 구조를 확립
   3. 콜라이더를 0으로 두지 않고 시각적 가이드를 유지함으로써 디자인 편의성을 챙기는 동시에, 게임 실행 시점에는 SO의 값만을 받아 정확한 데이터값이 적용되도록 하여 '데이터 무결성'과 '생산성' 을 동시에 확보한거야

   1. SlowTree를 첫 구현때는 AttackTree의 상속을 받으려 했으나 AttackTree는 여럿 타워 중 하나일 뿐, 중심이 되는 오브젝트가 아니기게 현재의 각자의 스크립트로 분리


**각 타워마다 사거리가 다를 수 있으니 각자의 SO를 추가하던지, 스크립트에서 SO값을 우선으로 처리, 혹은 So의 값을 쓰지 않고 스크립트에서 개별로 값을 추가**

LineRenderer를 활용하여 타워와 타겟 몬스터를 연결하는 조준선을 구현

- 타워가 적을 공격할 때, 사용자가 현재 어느 몬스터를 공격 중인지 직관적으로 알기 어려워 피드백이 부족해보였어. 
- 그래서 처음 사용해보는 LineRenderer 컴포넌트를 활용해 타워와 타겟 몬스터를 실시간으로 연결하는 시각화 로직 구현.
   1. AttackTree 스크립트의 Update() 함수 내에서 타겟이 존재할 경우 lineRenderer.enabled = true로 활성화
   2. Size는 2로 설정하고, SetPosition을 사용하여 0번 인덱스는 firePoint.position(조준선의 첫번째 위치), 1번 인덱스는 target.position(대상 몬스터의 위치)으로 매 프레임마다 업데이트
      2-1. LineRenderer 컴포넌트의 Size란?: Size가 2라면: [시작점] → [끝점] 딱 한 구간만 그어져. (내가 지금 하는 방식), Size가 3이라면: [시작점] → [중간점] → [끝점] 이렇게 꺾인 선을 그릴 수 있어
      2-2. SetPosition란? LineRenderer가 "어디서부터 어디까지 선을 그릴지" 그 좌표(Position)를 직접 지정해 주는 함수야, SetPosition은 라인의 각 점들을 내가 원하는 좌표로 실시간으로 이동시키는(갱신하는) 명령어라고 생각하면 돼)
         2-a. lineRenderer.SetPosition(0, firePoint.position);//0번 점(시작점)을 타워의 발사 위치로 옮겨!
         2-b. lineRenderer.SetPosition(1, target.position);   //1번 점(끝점)을 몬스터의 현재 위치로 옮겨!
         2-c. 만약 SetPosition을 안 쓰면 어떻게 될까? LineRenderer는 처음에 설정된 그 고정된 위치(아까 본 월드 중심 0,0,0)에만 선을 그려. 몬스터가 아무리 움직여도 선은 꼼짝도 안 하겠지, 
         하지만 Update() 함수 안에서 매 프레임 SetPosition을 호출하면 SetPosition(1, target.position) 코드가 몬스터의 새로운 좌표를 읽어와서 라인의 끝점을 그곳으로 이동시키고 마치 선이 몬스터를 계속 따라다니는 것처럼 보이는 거지
   3. 타겟이 사라지거나 범위 밖으로 나가면 else { lineRenderer.enabled = false; } 로 비활성화하여 리소스 낭비 방지

PauseManager 스크립트의 UI 연결 방식 최적화 및 안정성 확보

- GameObject.Find를 통한 자동 연결 방식이 비활성화된 UI 오브젝트를 탐색하지 못하는 이슈 발생!!
   1. 원인: 유니티의 GameObject.Find는 씬에 활성화된 오브젝트만 탐색 가능하다는 제약 사항이 있음 
   2. 아마 어제 난 PausePanel을 인스펙터에 드래그 연결을 한 상태에서 실행을 했기에 문제가 없다고 판단했나봐
- 오류가 뜨지 않았던 이유?: GameObject.Find("이름")로 비활성화된 UI를 찾을때 
   1. 찾기 시도: 유니티가 씬을 뒤져서 "PausePanel"이라는 이름을 찾으려고 해
   2. 못 찾음: 비활성화된 오브젝트라서 검색 결과에 포함되지 않아. 그럼 GameObject.Find는 null을 반환해
   3. 조용한 실패: 유니티는 null을 반환하고는 아무 일도 일어나지 않아. 오류를 내지 않는 이유는 "이름이 없는 걸 찾으라고 했다고 해서 그게 당장 프로그램이 아예 멈추진 않아" 라고 판단하기 때문이었어
   4. 그다음: 코드의 if (panelObj != null) 부분에서, panelObj가 null이니까 if 블록 안으로 들어가지 않았던 거야
   5. 즉, 오류가 안 난 게 아니라, if (panelObj != null)로 방어 코드를 짜두었기 때문에 코드가 안전하게 그냥 '건너뛴' 거지!!	
      5-1. Start() 안에서: panelObj가 null 상태니까 pausePanel 변수에 null이 할당되고, SetActive(false) 명령은 실행조차 안 됐어
      5-2. TogglePause() 안에서: pausePanel이 null이니까 if (pausePanel == null) return; 이 조건문에 걸려서 아무것도 안 하고 함수가 종료된 거지 내 코드가 오류를 알리지 말라고 한거야!

- 비활성화 객체 탐색 로직으로의 변경.
   1. Start() 함수에서 Resources.FindObjectsOfTypeAll<Canvas>()를 사용하여 씬 내의 캔버스를 탐색
   2. Transform.Find를 통해 비활성화 상태인 PausePanel을 직접 탐색하여 연결하도록 로직 변경
- 자동 연결 구조 완성:
   1. 인스펙터에서의 수동 드래그 앤 드롭 방식(하드코딩된 참조)을 **절대** 지양하고, 시스템이 런타임에 오브젝트를 스스로 찾아 연결하는 자동화 구현
   2. 씬 로드 및 초기화 과정에서 UI 탐색 로직이 안전하게 동작하도록 구조화

- 오늘의 트러블슈팅: GameObject.Find의 사용으로 인한 비활성화 된 UI 찾기 실패
   1. 원인: GameObject.Find가 비활성화된 객체를 찾지 못하니 NullReference 오류 발생
   2. 해결: Resources.FindObjectsOfTypeAll를 활용해 비활성화 상태의 자식 오브젝트까지 탐색 범위 확대하여 해결

- '자동 연결'이라는 편리함을 얻기 위해서는 유니티의 오브젝트 탐색 원리를 깊이 이해해야 해. 2D작품때도 고생했던 문제였는데도 말이지....GameObject.Find와 같은 기본 함수들의 제약 조건을 정확히 숙지해둬
- 자동 연결 로직을 구현하는 과정에서 코드 양은 늘어났지만, 시스템의 신뢰성은 크게 향상된거야. 오늘 아무것도 안한거랑 다름이 없다고 느껴졌지만 사실상 '자동 연결' 실패 상태를 바로 찾아내 버그를 수정한 큰 일을 한거야!

README에 `비활성화 UI 탐색 및 자동 참조 시스템 개선` 문제 해결 기재

- 어제 수정 완료했던 비활성화된 UI 찾기에 대해 적었어. 2D작품의 README의 문제해결 과정과 비슷하게 적었거든.
 
- 문제 상황:
   1. 기존 GameObject.Find() 방식은 비활성화된 오브젝트 탐색이 불가능하여 PausePanel 참조에 실패
   2. 수동 인스펙터 참조 방식은 씬마다 연결을 반복해야 하는 유지보수성 문제와 참조 누락(NullReference) 위험이 큼

- 분석 및 해결 과정.
   1. Resources.FindObjectsOfTypeAll<Canvas>()를 통해 비활성 객체까지 탐색 범위 확장
   2. scene.name 필터링으로 프리팹이 아닌 실제 씬 인스턴스만 정확히 선별
   3. Transform.Find("PausePanel")로 자식 UI를 찾아 동적 할당하여 하드코딩된 참조를 제거

- PauseManager 스크립트의 Start()함수의 핵심 코드를 추가.

- 회고 및 향후 과제.
   1. 유니티 API 활용: 모든 탐색 방식에는 장단점이 있으므로, 상황에 맞는 API를 선택하는 안목이 중요
   2. 방어적 설계: 에러를 숨기는 코드(if null)보다 에러를 빠르게 인지하고 해결하는 디버깅 습관이 필요
   3. 방법론의 한계 인식: 현재 방식은 프로젝트 규모가 커지면 이름 의존성으로 인해 한계가 있다. 향후 프로젝트 확장 시 DI(의존성 주입) 패턴이나 Tag/Layer 시스템으로 구조를 고도화할 계획
- 개발일지는 이렇게 [내가 겪은 문제 -> 해결 논리 -> 코드 증거 -> 스스로의 평가]가 들어있으면 나중에 3D 프로젝트를 완성했을 때, 너만의 훌륭한 '기술적 증거'가 될 거야.
